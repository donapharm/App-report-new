# App Report trusted-device SSO — contract v2

## Security boundary

- App Sale remains the source of truth for its trusted devices.
- App Report never reads App Sale DB and never receives the raw App Sale device id.
- `appsale_device_id` remains a **host-only** cookie on `sale.donapharm.asia`; never set `Domain=.donapharm.asia`.
- Missing, malformed, or shorter-than-32-byte `INTERNAL_TRUSTED_DEVICE_KEY` keeps both endpoints at HTTP 404. Missing/malformed `TRUSTED_DEVICE_REPORT_S2S_TOKEN_HASH` also keeps `consume` at 404.
- The key stays only in App Sale server configuration. Do not send it to browser, chat, logs, or source control.

## Report integration flow

### 1. Create a pending login

Report backend creates and stores:

- `nonce`: cryptographically random base64url, 22–128 characters;
- `reportDeviceId`: Report's own random device id, 16–200 characters;
- `expectedEmployeeCode`: canonical Report account code;
- expiry ≤ 2 minutes and status `pending`.

### 2. Browser requests the App Sale assertion

From exactly the configured `TRUSTED_DEVICE_REPORT_ORIGIN` (default `https://report.donapharm.asia`):

```js
const response = await fetch(
  "https://sale.donapharm.asia/api/internal/trusted-device/verify",
  {
    method: "POST",
    credentials: "include",
    headers: { "content-type": "application/json" },
    body: JSON.stringify({
      phone,
      expectedEmployeeCode,
      reportDeviceId,
      nonce,
    }),
  },
);
```

The browser sends App Sale's host-only cookie only to App Sale. App Report JavaScript cannot read it.

Successful trusted response:

```json
{
  "trusted": true,
  "assertion": "v1.<payload>.<hmac>",
  "assertionExpiresAt": "<ISO timestamp, 60 seconds>",
  "employeeCode": "DN001",
  "reverifyDays": 30
}
```

Untrusted/missing/duplicate cookie returns `200 {"trusted":false}`. Never treat timeout, 4xx, 5xx, malformed JSON, or `trusted:false` as success.

The assertion is bound to:

- issuer `app-sale`;
- audience `app-report`;
- canonical subject/employee code;
- SHA-256 of `reportDeviceId`;
- request `nonce`;
- unique `jti`;
- 60-second `iat`/`exp` window.

### 3. Report backend consumes the assertion once

Browser passes the assertion to Report backend. Report backend must load the matching pending login and call:

```http
POST https://sale.donapharm.asia/api/internal/trusted-device/consume
Content-Type: application/json
Authorization: Bearer <dedicated Report backend token>

{
  "assertion": "v1.<payload>.<hmac>",
  "nonce": "<stored pending nonce>",
  "reportDeviceId": "<stored Report device id>",
  "expectedEmployeeCode": "<stored canonical account code>"
}
```

Only `200 {"valid":true,...}` permits Report to create its own session. The consume endpoint authenticates the Report backend, then checks HMAC, TTL, issuer, audience, subject, Report device binding, nonce, and one-time `jti`. A second consume returns HTTP 409. The raw backend token stays only in Report's secret configuration; App Sale stores only its prefixed SHA-256 hash.

Report must atomically mark its pending nonce used before issuing the session and must also reject a reused/expired pending nonce locally. Do not trust the browser's phone/account fields during consume; use values stored by Report backend.

## Fail-closed matrix

| Condition | Result |
|---|---|
| App Sale key missing/weak | 404 |
| Report S2S token hash missing/malformed | consume 404 |
| Report S2S bearer missing/wrong | consume 401 |
| Origin is not configured Report origin | 403 / no readable CORS response |
| App Sale cookie missing or duplicated | `trusted:false` |
| Device expired/revoked/inactive/fingerprint mismatch | `trusted:false` |
| Phone ↔ App Sale user ↔ employee mismatch | `trusted:false` |
| Expected employee differs | `trusted:false` |
| Nonce reused at assertion issue | 409 |
| Assertion tampered/expired/wrong binding | 403 |
| Assertion consumed twice | 409 |
| Timeout/429/500 | Report keeps OTP login; never fallback-open |

## Operations

Set via the approved secret/config channel only:

```dotenv
INTERNAL_TRUSTED_DEVICE_KEY=<base64 generated by openssl rand -base64 48>
TRUSTED_DEVICE_REPORT_S2S_TOKEN_HASH=<sha256 of `trusted-device-report-s2s:` plus a separate random token>
TRUSTED_DEVICE_REPORT_ORIGIN=https://report.donapharm.asia
TRUSTED_DEVICE_RATE_LIMIT_PER_MINUTE=60
```

Generate a separate random Report token and calculate the lowercase hex digest of the exact UTF-8 string `trusted-device-report-s2s:<token>`. Store the raw token only in Report's backend secret configuration and only the digest in App Sale. Never reuse `INTERNAL_TRUSTED_DEVICE_KEY` as this token.

Key rotation immediately invalidates outstanding assertions (maximum lifetime 60 seconds). Migration `0103_trusted_device_report_replay.sql` is required before enabling the bridge; it stores only short-lived hashes/IDs and provides atomic replay protection across restart/multiple API instances.
